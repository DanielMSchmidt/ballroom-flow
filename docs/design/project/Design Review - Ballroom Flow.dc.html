1 · Findings, by severity

Each: screen + frame reference · what's wrong · minimal fix. "Impl drifted" = code should change; "bundle stale" = wireframes/docs should change.
BLOCKER F1 — Reading view: ~8px interactive text, targets nowhere near 44px
Assemble · reading  ·  frames 1.6 1.13  ·  screenshot 01-fed.png  ·  impl drifted and bundle needs a scale note
The shipped view copies the wireframe's lo-fi pixel sizes literally: column headers (Step / Rise / Pos / Feet / Turn / Head), value chips, "+ add comment", "+1 more", and the STEPS FOR segmented control all render at ~8–9px with ~20–24px rows. All of these are tap targets — headers and chips open the info sheet (1.13), comment rows open threads (1.14), the toggle flips role. That breaks the system's own --bf-touch-target: 44px rule (#3), the rem type scale (#10), and WCAG 1.4.4 / 2.5.8. Root cause: "pixel-for-pixel parity" applied to a 320px-wide lo-fi frame — 8px there is shorthand, not a spec.
Fix: map wireframe px to token roles — chips + headers ≥ --bf-text-2xs (~12px), step rows min-height 44px with the whole cell as the hit area, header row 44px tall (small-caps text can stay small; the padded cell is the target), "+ add comment" becomes a full-width 44px row. With >5 kinds in play (Head, custom types) the grid horizontally pans with a sticky count column + edge fade, instead of shrinking type further. Add the scale-mapping note to the bundle (edit list, item 7).
BLOCKER F2 — Contrast failures: solid chips, muted labels, ghost actions
Assemble · reading + edit grid  ·  frames 1.6 1.11  ·  01-fed.png  ·  impl drifted (DS #6 pairings exist and are unused)
Shipped chips are solid kind-color with white text — bespoke styling that bypasses the documented -ink on -tint pairings. At chip sizes white fails AA on most kinds: rise #1f8a5b ≈ 4.3:1, footwork #a9742c ≈ 3.9:1, foot-position #2c8a85 ≈ 4.1:1, Head #0f8a8a ≈ 4.2:1 (small text needs 4.5). Secondary labels ("reading", "1ST LONG SIDE", "STEPS FOR" at ~#9a958c) ≈ 3.0:1; the "+ add comment" ghost (~#c0bab0 on #fafaf8) ≈ 1.9:1 — below even the 3:1 UI-component floor.
Fix: chips = --bf-<kind>-tint background + -border + -ink text (the tokens were chosen for exactly this); muted labels → --bf-ink-secondary; "+ add comment" → --bf-accent-ink at 12px. Registry-driven custom kinds derive tint/ink from their stored color — never solid bg + white.
SIGNIFICANT F3 — Parity: whole-figure notes moved above the step grid
Assemble · reading  ·  frame 1.6  ·  01-fed.png  ·  impl drifted
1.6 places the WHOLE FIGURE thread after the last step row; the shipped app puts it directly under the figure title, above the column headers — every annotated figure pushes its steps ~3 rows down. The bundle's order is deliberate (reading = "a clean programme for the floor": steps first, commentary after).
Fix: move the block back below the grid. If fast access to figure-level notes matters, add a small "2 ✎" count chip on the figure title row that scrolls to the thread — as a bundle addition, not by keeping the shipped order.
SIGNIFICANT F4 — Identity color is the only author signal on inline comments
Reading + thread  ·  frames 1.6 1.14  ·  01-fed.png  ·  both bundle and impl
Inline comment rows attribute authorship by a 7px colored dot alone — DS #5 ("color never the only signal — member identity colors") is violated by the canonical wireframe itself; the thread view (1.14) pairs color + initial + name, inline does not. A coach's correction vs. a partner's reply is a real distinction between rounds, and it's invisible to color-blind users.
Fix: replace the dot with a 16px identity chip carrying the author's initial (color + letter, aria-label = full name). Backport into 1.6's comment rows and the legend.
SIGNIFICANT F5 — "Made this figure yours" misteaches the copy-on-write model
Figure editor toast  ·  frame 1.12 (canonical toast)  ·  DS §3  ·  impl drifted + DS doc stale
Three competing strings exist: bundle 1.12 says "Copied into this choreo", DESIGN-SYSTEM.md §3 still says "Copied as your variant" (retired live-variant language), the app shipped "Made this figure yours". The app string is the worst of the three: "yours" implies personal ownership / personal library — but the copy is choreo-owned, edited jointly by everyone on the choreo, and it never entered your library. Microcopy here is the main way users learn the frozen-copy model.
Fix: standardize on "Copied into this choreo" (PLAN §4.9/§5.2 language) everywhere; optional second line "The library figure is untouched." No Undo action on this toast — cross-document COW undo is explicitly out of scope (Q-UNDO). Update DS §3's example. Spec in §2-C below.
SIGNIFICANT F6 — Undo/Redo in the figure editor: right feature, undesigned
Figure editor header  ·  frames 1.21 (Assemble only) 1.11 (caption mentions, frame doesn't draw)  ·  undesigned addition
Placement is right — a full-screen auto-saving editor needs undo in the header. What's missing is the spec: sizes, disabled states, ordering against ✎, and the toast contract. 1.21's toast also lacks the Redo action that DS §3 requires.
Fix (backport to 1.11): header order ‹ · [dot + title + context] · ↶ · ↷ · ✎; ↶/↷ are 44×44 IconButtons (plain variant, 20px glyph), accent #2f5d8f when enabled, ~40% ink-faint + aria-disabled when the stack is empty; a new edit clears redo (PLAN M1.9). Toast: "Undone — <change summary>" + [Redo] action; keep the shipped warning-tone "Undone — others had built on this change" for the superseded case. ⌘Z/⇧⌘Z at lg. Mock in §2-A.
SIGNIFICANT F7 — Library affordance: wrong surface on placement cards
Assemble placement cards + figure editor  ·  precedent: 2.1 2.2 ↟ save; PLAN §4.4 puts save on the timeline  ·  undesigned addition
On placement cards the button is clutter and a mis-tap hazard: cards already carry scope dot, name, badge, count chip and (editing) the ⠿ drag handle at the same trailing edge. "Save to my library" is a considered decision about reuse, which is exactly what the figure editor is for — and the bundle already gives the Library tab the ↟ pattern for the browse case.
Fix: remove it from placement cards. In the figure editor, a chip in the header meta row (not the title bar): "↟ Save to my library" (amber outline) ⇄ "✓ In your library" (amber tint, #6b4a16 ink). 32px tall, ≥44px hit with padding. Saved state is idempotent, not a toggle — tapping again toasts "Already in your library"; removal stays a deliberate Library-tab action. Toasts "Saved to your library" / "Already in your library" are already locked in PLAN §4.2. Mock in §2-A/B.
SIGNIFICANT F8 — "Used in N choreos": right instinct, wrong trigger
Figure editor  ·  precedent: 2.3 "used in N routines" on library cards  ·  undesigned addition
As shipped it warns that "editing a shared figure changes every choreography using it" — but under the reconciled model that is true in exactly one context: editing a personal-library figure in place (Library tab ✎, US-034). Editing from inside a choreo either edits a choreo-owned figure (this choreo only) or copies-on-write (changes nothing else). Showing the warning in the wrong context re-teaches the fear the frozen-copy model was chosen to remove.
Fix — three editor contexts: (1) personal-library figure, in-place: persistent banner under the header — "↟ In your library · used in 3 choreos — edits update all of them" (amber tint, ≥44px; count from the D1 projection; singular "1 choreo"). (2) choreo-owned figure: nothing — the default stays quiet. (3) global / outside figure: no warning; the header context line reads "global figure — editing copies it into this choreo", and the COW toast (F5) confirms on first edit. Mock in §2-B.
SIGNIFICANT F9 — DESIGN-SYSTEM.md's attribute-kind table is stale vs. the registry
DS §2  ·  PLAN §3  ·  bundle stale (docs side)
DS §2 documents five kinds and still models "step / footwork" as one amber kind ("Fw") — but the registry split direction (#2f5d8f) from footwork (#a9742c) in the 2026-06-28 notation work, and the UI ships footPosition (#2c8a85, the "Feet" column) and bodyActions besides. Any column the token table doesn't cover gets bespoke hex styling by default — this is how the F2 drift happens for every new kind.
Fix: regenerate DS §2 from the merged ATTRIBUTE_REGISTRY — rows for direction / footwork / footPosition / rise / position / bodyActions / sway / turn with tint/ink/border tokens, header vocabulary matched to the UI (Step · Rise · Pos · Feet · Turn), and the derivation rule for user-defined kind colors.
SIGNIFICANT F10 — Parity evidence gap: only one shipped screen was attached
Process  ·  screenshots/ + uploads/
Of the ~10 surfaces in scope, only the reading view has current shipped evidence (three identical captures). PLAN §14 declares full parity as of 2026-06-29 — that claim is unaudited. Focus visibility and reduced-motion also cannot be judged from stills.
Fix: capture the matrix — choreo list, editing, edit grid, attribute editor, info sheet, thread, library ×2, journal ×3, link picker, share, profile, overlays, toasts — including the bundle's own empty / few / many states, plus one keyboard-focus and one prefers-reduced-motion pass. I've limited screen-level parity findings to what the evidence supports.
MINOR F11 — DS prescribes a switch for the Leader/Follower view
DS §3 maps the role flip to Toggle (role="switch"), but the designed control (1.6/1.11) is a two-segment labeled choice. A switch reads as on/off — it would make one role "off". The shipped segmented look is right; the doc is wrong. Fix: DS → segmented pair (aria-pressed Chip pattern or Tabs), 44px tall.
MINOR F12 — "routines" vs "choreos"
The tab says Choreo; cards, quota and library copy say routines ("used in 2 routines", "3 routines on the free plan"); the new indicator says choreos. Fix: one user-facing noun — recommend "choreos" (matches the tab and the product voice); sweep bundle 2.3 / 1.18, DS §3 toast, and app strings. Keep "routine" for PLAN-internal prose only.
MINOR F13 — Header action drift in reading
Frame 1.6: quiet ✎ and ↗ glyphs. Shipped: filled circular buttons and a share-nodes icon. Either is defensible; pick one — recommend DS IconButton plain at 44px to match the bundle's quiet header, and update the losing side (covers F17's share-glyph mismatch too).
MINOR F14 — Bundle bug: undefined var(--note-font) in 1.17
1.17's caption sets font:500 13px var(--note-font); the variable is defined only in the Builder prototype, so the line silently falls back to the default font. Fix: replace with Caveat; grep the v4 file for other Builder leftovers.
NIT F15 / F16 — small polish
F15: shipped column headers are underlined — at 8px they read as desktop hyperlinks; when resized per F1, drop the underline (the header tap itself is the info affordance per 1.13).  F16: bundle hint text below contrast ("tap a cell to add / edit" #a9a299, various #b0aaa0 lines) — map to --bf-ink-muted when backporting.

1b · User-testing feedback — triage

Four observations from testing. Two corroborate findings above; two are new (U2, U3).
CORROBORATES F1+F2 U1 — "Didn't see the add-comment parts in the choreo overview"
Expected: an 8px ghost at ~1.9:1 contrast is functionally invisible. The F1/F2 fix (full-width 44px row, --bf-accent-ink, 12px) is validated — ship it first and retest before adding anything louder. If it still misses, the fallback is a leading ＋ icon in a 20px accent-tint circle on the row, not more color.
NEW · SIGNIFICANT U2 — Attribute explanation: sheet → full page
Frame 1.13 redesign · bundle change
Users want the kind-explainer (tap a column header / chip) as a full page you can go back from, not a bottom sheet — same open mechanics, more room to read.
Fix: rework 1.13 as a full-screen sub-page: header ‹ · [kind dot + name] (44px back target), then the explainer content — what the kind encodes, the value vocabulary with tinted chips at readable size, and an example row. It gets its own route so hardware/gesture back works, and state (scroll position in the grid) is preserved on return. Tapping a specific chip deep-links to that value's entry. Keep bottom sheets for actions (link picker, attribute editor); explanation is reading material and earns a page.
NEW · SIGNIFICANT U3 — Greyed half-beat timings read as "disabled"
Frames 1.6 1.11 count column · bundle + impl
Non-full-beat counts ("&", "a") are rendered in faint grey to de-emphasize them — but grey is the UI's disabled vocabulary, so users read the "&" steps as inactive or somehow broken. Quick–quick–& timing is core content, not chrome.
Fix: differentiate by weight and size, never by fading: full beats bold at the base size, sub-beats regular at one step smaller, both in full --bf-ink; optionally set sub-beat rows on a subtle accent-tint band to group them under their beat. Reserve grey strictly for disabled. Backport to the count column spec in 1.6/1.11 and DS §2.
EXTENDS F7 U4 — "Unclear where a saved figure lands"
"Saved to your library" confirms the action but not the destination — users don't yet have a mental model of the Library tab › Mine shelf. Fix, three parts: (1) the toast gains a destination + action: "Saved to Library › Mine  [View]" — View jumps to the Library tab, Mine filter, new figure highlighted for ~2s; (2) the ✓ chip label becomes "✓ In My Library" matching the tab's shelf name exactly (one vocabulary, F12 spirit); (3) first save only: the Library tab icon pulses once (respecting --bf-motion-*). Backport the toast to 1.11/2.1 and PLAN §4.2.

4 · (b) Top 3 changes for the daily "notate after practice" flow

1
Make the reading grid legible at arm's length (F1 + F2 + F3): 44px rows, 12px tinted chips, steps before notes, sticky count column when columns overflow. This is the surface dancers check between heats — right now it's an 8px squint test with sweaty hands.
2
One-thumb capture: full-width 44px "+ add comment" rows, plus a sticky "＋ note" action on the reading view that pre-anchors to the figure currently in view (then hands off to the existing thread / link flow). Between rounds you have ~60 seconds and one thumb; today capture costs a precision tap on an 8px ghost.
3
Make editing feel safe (F5 + F6 + F8): visible undo/redo, "Copied into this choreo", and the ripple banner only where ripples are real. Confidence that a quick fix can't wreck the shared routine is what gets notation done at practice instead of at home.

5 · User-testing feedback — diagnosis & fixes

Four observations from testing, each mapped to a root cause and a backportable fix.
USER U1 — "I didn't see where to add a comment" (reading view)
Assemble · reading  ·  frame 1.6  ·  confirms F1/F2's ghost-row finding empirically
Predictable: the "+ add comment" row is an 8px ghost at ~1.9:1 contrast (F2) that only trails some figures, and it competes with the visually louder step grid. Raising size/contrast alone won't fully fix it — a per-figure inline row also can't express "comment on step 2".
Fix (layered): (1) restyle the row per F1/F2 — full-width 44px, ✎ glyph + "Add note" in --bf-accent-ink, hairline top border so it reads as part of the figure. (2) Long-press any step row → "Add note on step N" (matches the edit grid's cell-tap mental model; spec in 1.13's interactions). (3) The sticky "＋ note" action already proposed in §4-2 covers the "just capture something" case. Test again with (1)+(3) before adding chrome.
USER U2 — Attribute explanations: sheet → full page
Explanation surface  ·  frame 1.13  ·  bundle change requested by users
Users want the same triggers (tap a column header or value chip) but a full page they can go back from, not a bottom sheet. Reasonable: the content is educational reading (definitions, value tables, examples), not a quick confirm — sheets suit transactions, pages suit reading. A page also gives room for the per-value list that 1.13 currently truncates.
Fix: replace 1.13 with an Attribute explainer page: pushed navigation (‹ back returns to the exact scroll position in reading), header = kind chip + name ("Rise & fall"), body = one-line definition, value table (each value chip at full size + meaning), and "in this choreo" examples linking to their steps. Tapping a value chip opens the page scrolled to that value, highlighted. Keep it one page per kind with prev/next kind arrows in the footer (Step ‹ › Rise…) so curious readers can browse. Deep-linkable route (/choreo/:id/kinds/:kind) so back/forward and PWA history behave. Bundle: redraw 1.13 as a full frame, update the ⓘ interaction note in 1.6.
USER U3 — Dimmed off-beat timings read as "disabled"
Reading + edit grids  ·  frames 1.6 1.11  ·  bundle + impl share the pattern
The bundle renders & / a sub-beat rows "dimmed between the counts" (grey ink on grey fill) — the same visual code the whole system uses for disabled/faint states, so users read them as inactive rows rather than faster beats. In a Weave ("4 & 5 & 6 a") half the figure looks switched off. Dimming also collides with F2's contrast floor.
Fix: encode "sub-beat" by structure, not opacity — full-contrast ink for the count label and attributes; differentiate with (a) a narrower/indented count cell nested under its parent beat (the "4" cell visually brackets the "&"), (b) a smaller count-label size one type step down, and (c) the step-accent tint at half strength as row fill — never grey. Reserve grey fill + faint ink exclusively for genuinely empty/disabled cells. Backport to 1.6, 1.11 and the "off-beat timings" caption; add the rule to DS §2 ("dimming = disabled, never = subdivision").
USER U4 — "Where did my saved figure go?"
Save-to-library flow  ·  frames 2.1 2.3  ·  extends F7
The save action gives a toast ("Saved to your library") but no place: "library" is ambiguous between the global catalog the user just browsed and their personal shelf, and nothing on screen changes. Classic invisible-destination problem — the F7 chip flip helps state, not location.
Fix: (1) toast names the destination and takes you there: "Saved to My figures · [View]" — View opens the Library tab, "My figures" filter active, the new card highlighted for ~2s. (2) Name the shelf "My figures" consistently (tab segment in 2.1, chip copy becomes "↟ Save to My figures" / "✓ In My figures", banner in F8 follows) — "library" stays the tab name only. (3) One-time badge dot on the Library tab icon after a first-ever save. Bundle: update 2.1's segment labels, F7's chip copy in the new 1.11 meta row, and the PLAN §4.2 toast strings.
Not restated at length: the v4 bundle's information architecture, state coverage (empty/few/many), the frozen-copy model's screen mapping, and the token/primitive layer are sound — the findings above are the gap between that intent and what shipped or got added ad hoc. Tweaks panel: filter by severity floor, hide mockups (for a text-only print).
